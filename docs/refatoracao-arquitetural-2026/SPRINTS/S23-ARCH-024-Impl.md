- **Decisão sobre etapa 4 (ambiente real) — autocorrigida durante a própria Sprint:** cheguei a registrar "dispensada", com a justificativa de que a mudança é puramente estrutural (aninhamento de campos já existentes, sem lógica nova) e que `tsc --noEmit` já garante que todo ponto de leitura/escrita foi atualizado de forma consistente. Revertido ao reler a Regra Permanente #5 deste documento: a Classificação do card ARCH-024 é `Exige RFC`, e a regra torna a etapa 4 obrigatória categoricamente para essa classificação antes de fechar `🟢 Concluída` — não é uma avaliação de risco caso a caso que eu possa substituir pelo meu próprio julgamento. Além disso, identifiquei um risco real que a alegação de "puramente estrutural" tinha subestimado: todo campo de `DeliveryTrackingContext` é opcional (`?.`), então um erro de wiring (aninhamento incorreto, produtor esquecendo de propagar o sub-objeto) não geraria erro de tipo — apenas faria os guards de dedup virarem no-op silencioso, reintroduzindo exatamente os bugs de envio duplicado (S10/S44/S51) que este subsistema existe para prevenir. `tsc` não cobre esse risco porque `undefined` é um valor válido para um campo opcional.
