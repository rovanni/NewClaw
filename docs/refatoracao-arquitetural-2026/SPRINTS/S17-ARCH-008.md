- **Arquivos afetados:** nenhum tocado em código — só documentação (a Sprint foi adiada na Fase 1, antes de qualquer mudança).

- **Premissa reverificada (Fase 1) — não se sustentou, categoria nova (6º modo de falha):** o card justificava a mudança citando "o sistema já tem recovery de goals ativos no boot" — **esse mecanismo não existe**. `AgentController.getAllActive()` roda no boot e no shutdown só para LOGAR (`recovered=false` explícito no próprio log de produção) — nunca chama `resumeGoal()`/`runLoop()`. Um goal deixado `executing` quando o processo morre fica permanentemente órfão, sem qualquer recovery, com ou sem este ARCH.
- **O defeito subjacente é real, só o gatilho é outro:** o único call site de `resumeGoal()` é `GoalOrchestrator.resumeFromAuth()` (aprovação de ação perigosa) — mesmo processo, sem restart algum. `progressModel` reseta ali igualmente, e esse caminho é MAIS frequente/fácil de reproduzir do que o cenário de restart que o card descrevia.
- **Fix desenhado (não implementado):** `buildInitialProgressModel(goal)` — reconstrói `components` a partir de `goal.currentPlan` (steps `completed`) + `goal.attempts` (steps `pending` com attempt já registrado), no mesmo espírito de `buildIncrementalExecutionContext()`. Usado só na criação do `state` em `runLoop()`. Para `executeGoal()` (goal novo), retorna `[]` — zero mudança nesse caminho. Registro técnico completo: `docs/issues/009-arch008-no-automatic-boot-recovery-exists.md`.
- **Decisão do usuário:** adiar e documentar, mesma linha de S14 — a premissa errada era estrutural o suficiente (o mecanismo citado como motivação simplesmente não existe) para justificar não implementar sem revisão adicional, mesmo com o fix já desenhado.

## Reabertura e implementação — 2026-07-19 (3º item do próximo ciclo, após ARCH-009/ARCH-013)

- **Fase 1 (reverificação):** premissa de julho confirmada, e mais forte que antes — `runLoop()`
  ainda cria `state.progressModel` vazio incondicionalmente, mas um comentário no código (fix de
  vazamento entre goals concorrentes, Sprint 0.6 Front A) revela que isso virou uma garantia
  estrutural deliberada, não um acidente. `AgentController.getAllActive()` continua log-only;
  `GoalOrchestrator.resumeFromAuth()` continua o único call site real de `resumeGoal()`.
- **Fase 2 (simplificação do design de julho):** o fix desenhado propunha re-escanear
  `goal.attempts` manualmente para derivar status de steps `pending` já tentados — obsoleto, porque
  `ARCH-007` (S13) já adicionou `PlanStep.lastAttemptOutcome` (sinal restart-safe, já persistido).
  Reescanear seria uma 2ª fonte da mesma informação — corrigido para ler `lastAttemptOutcome`
  diretamente.
- **Achado de consequência real (não só cosmético):** `state.progressModel.overallPercent`
  alimenta a lógica `ADAPTIVE-BUDGET` (bônus de +1 replan quando progresso real >=60% e
  `replanBudget` esgota) — `GoalExecutionLoop.ts` ~linha 1413. Com o bug, todo goal retomado via
  `resumeGoal()` perdia esse bônus, porque `overallPercent` sempre lia 0%, mesmo com progresso
  real substancial — não é só uma barra de progresso visual incorreta.
- **Implementado:** `buildInitialProgressModel(goal): GoalProgressModel` novo — deriva componentes
  de `PlanStep.status`/`PlanStep.lastAttemptOutcome`; fórmula de `overallPercent` espelha a já
  usada em `adoptNewPlan()` (`completed / (completed + pendentes do plano atual)`). Usado na
  inicialização de `state` em `runLoop()`. Para `executeGoal()` (goal novo): `components=[]`,
  `overallPercent=0` — idêntico ao literal fixo anterior, zero mudança nesse caminho.
- **Teste novo:** `S127_ARCH008_BuildInitialProgressModel.test.ts` — 9 asserções unitárias (goal
  novo, 1 completed + 1 pending sem attempt, 1 completed + 1 pending com `lastAttemptOutcome`
  → `in_progress`, e o cenário real do bug: 3/4 steps completed preserva >=60%).
- **Checklist de validação — Validação Progressiva completa (4 etapas):**
  1. Unitário: `S127` (9/9).
  2. Regressão completa: **130/130** (129 pré-existentes + `S127` novo, 0 quebras).
  3. `tsc --noEmit`: limpo.
  4. **Ambiente real** — achado importante do processo: o Web Dashboard (único canal configurado
     no sandbox) **não tem `workflowCallback` wired** (`ARCHITECTURE.md` já documenta isso — só
     Telegram/Discord/WhatsApp/Signal reconhecem `auth:approve:<txnId>`), então não existe caminho
     HTTP real para aprovar uma ação perigosa via Dashboard. Validação real feita em 2 metades:
     (a) goal real via HTTP, LLM real, atingiu `needs_auth` genuíno de verdade — log real
     `[AUTH] Dangerous tool BLOCKED: exec_command`, `pendingTxnId` real persistido em SQLite;
     (b) `GoalOrchestrator.resumeFromAuth()` chamado diretamente contra o MESMO banco real, via
     `AgentController` real (mesma classe/wiring de produção, só sem abrir HTTP/canais) — log real

     verdade (`GOAL-RESULT success=true`). Gap do canal Web Dashboard registrado como achado
     lateral, não é bug desta Sprint — mesma categoria dos gaps de anexo WhatsApp/Signal já
     documentados em `ARCHITECTURE.md`.
