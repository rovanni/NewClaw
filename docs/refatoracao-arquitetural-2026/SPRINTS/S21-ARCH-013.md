## Reabertura e implementação — 2026-07-18 (2º item do próximo ciclo, após ARCH-009/S14)

- **Gatilho:** auditoria explícita a pedido do usuário, mesma rodada que reabriu S14.
- **Fase 1 (reverificação):** premissa de julho continua correta — confirmado por `git log` que
  nada tocou `evaluateAgentStepSuccess`/`escalateStepEvalToLLM`/`StepSemanticValidator` desde o
  adiamento, mas S22/S24 REORGANIZARAM onde esse código vive (`dispatchAgentloopStep()`,
  `finalizeStepAttempt()`, `runValidationPhase()` — extraídos de `executeStep()`/
  `runLoopInternal()` monolíticos que existiam em julho). A leitura do código pós-S22/S24 revelou
  um problema mais grave que o notado em julho: remover `escalateStepEvalToLLM` sem mais nada
  faria `evaluateAgentStepSuccess` retornar `success:false` pra zona ambígua (não só
  `stepSuccessConfident:false`) — o que faria `cycleResult.outcome` nunca chegar a `'success'`,
  e portanto o guard que dispara `StepSemanticValidator.validate()`
  (`if (cycleResult.outcome === 'success' ...)`) NUNCA rodaria pra nenhum step ambíguo — a própria
  correção proposta em julho se autoanularia.
- **Achado que destravou a implementação:** `GoalStore.ts` já tinha `downgradeLastAttemptToPartial()`
  e `finalizeLastAttemptAsSuccess()` — um padrão read-modify-write já testado em produção pra
  corrigir retroativamente um `GoalAttempt` já persistido, exatamente o mecanismo que faltava pra
  viabilizar a promoção sem reestruturar a ordem de dispatch. `GoalStore.promoteLastAttemptToSuccess()`
  novo reusa a mesma base (`updateLastAttempt`), predicate `result === 'partial'`.
- **Design implementado (Fase 3, revisado 2x durante a própria implementação):**
  1. `evaluateAgentStepSuccess()` simplificada: os ramos "zona ambígua 15-200 chars" e "resposta
     longa sem sinal" (que já existiam separados, um escalava LLM, o outro nunca podia ser
     promovido) foram fundidos num só — sempre `success:true, confidence:0.70`, nunca confiante.
     `escalateStepEvalToLLM()` removida inteira.
  2. `StepSemanticValidator` ganhou `shouldPromoteToConfidentSuccess` — contraparte positiva de
     `shouldDowngradeToPartial`, computada nos 2 branches (fast path e slow path via LLM).
  3. **Threshold de promoção é `FAST_PATH_CONFIDENCE_THRESHOLD` (0.72), não o `0.80` usado por
     downgrade** — decisão deliberada, não descuido: um teste de bancada (write-up completo
     abaixo) mostrou que usar 0.80 criaria uma "zona morta" de confiança fast-path (0.72-0.80)
     onde o step já pulava a chamada de LLM (por já ser confiante o bastante pra não precisar
     perguntar) mas não era confiante o bastante pra promover — ficando `'partial'` pra sempre,
     pior que o comportamento anterior (que sempre tinha uma chance de LLM confirmar). Reusar o
     próprio threshold do fast path elimina essa zona morta por definição.
  4. `runValidationPhase()` ganhou um `else if (shouldPromoteToConfidentSuccess)` entre o branch de
     downgrade e o de log-only-mismatch, chamando `promoteLastAttemptToSuccess()`.
- **Achado colateral corrigido (não escondido):** 2 testes de regressão pré-existentes
  (`S85_PartialAttemptResult_LowConfidenceSuccess`, `S119_PlanStepLastAttemptOutcome_ARCH007`)
  usavam texto de fixture com overlap de palavras-chave DELIBERADO com a description do step (pra
  não disparar um downgrade falso-positivo antes desta Sprint) — esse mesmo overlap agora também
  dispara PROMOÇÃO, então 3 dos cenários antigos (que testavam `escalateStepEvalToLLM`, removida)
  ficaram obsoletos e foram redesenhados pra testar os 2 caminhos novos (fast path 0 chamadas de
  LLM / slow path com LLM real) em vez do mecanismo antigo. `S119.2` teve só o texto de fixture
  ajustado (sem overlap) pra continuar testando o que sempre testou (propagação de
  `lastAttemptOutcome`), não o mecanismo de promoção em si.
- **Teste novo:** `S126_ARCH013_SemanticPromotionReplacesEscalation.test.ts` — 11 asserções,
  unitário sobre `GoalStore.promoteLastAttemptToSuccess()` (promove só quando `result==='partial'`,
  no-op caso contrário, preserva `output`) e `StepSemanticValidator` (fast path promove, slow path
  promove, `'relevant'` com confidence abaixo do threshold NÃO promove — pina o valor exato 0.72 —,
  `'mismatch'` nunca promove).
